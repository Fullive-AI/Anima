# Anima 项目分享 — 架构概览与演讲稿

---

## 一、一句话定位

> **Anima 是一个运行在你本地的硬件 AI 操作系统。**
> 它自动发现局域网里的智能设备，给每台设备装上专属 AI 技能，让它们能自主感知、思考、行动——不是"用手机 App 遥控开关"，而是"设备自己懂得该做什么"。

### 三层设计哲学

| 层次 | 组件 | 职责 |
|------|------|------|
| **快速通道** | 规则引擎 | 毫秒级响应，无需 LLM，处理紧急情况（温度超标→立即开空调） |
| **智能通道** | LLM 大脑 + 技能 | 加载领域知识，调用 AI，输出结构化 JSON 动作 |
| **记忆通道** | 记忆体 | 存偏好、存历史，让 AI 决策越来越懂你 |

## 三、最有价值的设计——技能系统

传统智能家居：遥控开关。
Anima：给每类设备装"领域大脑"。

```
skills/humidifier/
  skill.yaml        # 声明：这个技能适用于 humidifier 类型设备
  knowledge.md      # 领域知识：舒适湿度 40-60%，夏季偏低，冬季配合暖气...
  actions.py        # 可执行动作：set_humidity, set_mode, turn_on/off
  prompts/
    decide.md       # LLM 提示词模板：当前数据 + 用户偏好 + 历史 → 输出 JSON
    learn.md        # 学习提示词：每天总结一次，更新偏好文件
```

**AI 决策的实际流程：**

```
传感器上报湿度: 28%
        ↓
规则引擎: 28% < 20%？不触发（未超阈值）
        ↓
事件总线: 触发 SENSOR_UPDATED 事件
        ↓
LLM 大脑:
  拼装上下文 = 当前数据(28%) + 知识库(舒适40-60%) + 用户偏好 + 最近历史
  调用 LLM → 解析 JSON 响应:
  { "action": "set_humidity", "params": {"value": 50}, "reason": "当前28%低于舒适值" }
        ↓
执行动作 → 记录历史 → Dashboard 实时刷新
```

## 四、现有功能清单（v0.1）

### 后端
- ✅ 异步事件总线（支持通配符订阅）
- ✅ 规则引擎（3 条默认规则：高温/低湿/高湿触发器）
- ✅ LLM 大脑（langchain-openai，支持 GPT / DeepSeek / 豆包 / Ollama）
- ✅ 技能系统（加湿器 / 空调 / 灯光 / 协调者 4 个内置技能）
- ✅ 记忆体（preferences.md + history.json + learned.md，纯文本，无数据库）
- ✅ MIoT 适配器（小米设备云端发现 + 局域网 UDP 扫描）
- ✅ 调度器（每 5 分钟扫描设备，每天学习偏好）
- ✅ REST API（FastAPI，8080 端口，8 个接口）
- ✅ MQTT 消息总线（amqtt，无需 Docker）

### 前端 Dashboard
- ✅ 三栏布局：设备列表 | 设备卡片/传感器 | AI 决策流
- ✅ 实时设备状态显示
- ✅ AI 决策流时间线（显示每次决策的动作 + 原因）
- ✅ 底部聊天栏（和 Anima 对话）
- ✅ 设置面板（LLM 配置 / 小米扫码登录 / 手动添加设备）

## 五、演讲稿

### 开场（1 分钟）

"我今天分享的项目叫 **Anima**，拉丁语里是'灵魂'的意思。

我们平时用小爱同学、米家 App 控制智能设备，本质上还是**人在操作机器**——你说开灯它才开，你说调湿度它才调。

Anima 想做的事情是反过来：**机器主动感知环境，自己决定该做什么**。不是简单的'湿度低于40%开加湿器'这种固定规则，而是真正理解：现在几点、空调开没开、你昨天的偏好是什么——然后给出一个聪明的决定。"

---

### 架构讲解（3 分钟）

"整个系统是一个**单进程的 Python 后端**加上一个**React 前端**。

核心有三条处理通道——

**第一条：快速通道**。规则引擎，不走 AI，毫秒级响应。比如温度超过 35 度，直接触发开空调，这类紧急情况不需要思考。

**第二条：智能通道**。这是 Anima 的核心特色——**技能系统**。每种设备类型有一个专属技能包：里面有领域知识文件，告诉 AI 加湿器的舒适范围是 40%-60%、冬天要配合暖气调高湿度……有提示词模板，把实时传感器数据、用户历史偏好、领域知识拼在一起送给 LLM，LLM 输出 JSON 格式的动作指令，系统解析后执行。

**第三条：记忆通道**。AI 每天总结一次你的使用习惯，存成 Markdown 文件——不用数据库，直接可读。下次决策时把这份偏好上下文带进去，模型会越来越懂你。

设备层通过 MQTT 消息总线通信，现在已实现小米/米家设备的适配器，支持云端发现和局域网扫描两种方式。"

---

### Demo 演示（3 分钟）

"我们打开控制面板——

**[指向左侧]** 这里是设备列表，显示当前发现的所有设备。

**[指向中间]** 选择一个设备，这里显示它的实时传感器数据——温度、湿度、电源状态。

**[指向右侧]** 这里是 AI 决策流，每次 AI 做完决策，动作和原因都会实时出现在这里。比如这条：'当前湿度 28%，低于用户偏好 50%，执行 set_humidity(50)'。

**[点击设置]** 设置面板里可以配置 LLM——支持 OpenAI、DeepSeek、本地 Ollama 等任何兼容 OpenAI 格式的 API。还可以扫码登录米家账号，自动拉取你所有设备的控制 Token。

**[底部聊天栏]** 这里后续版本会支持自然语言控制：'把卧室调暖和点'，AI 自动推断动作。"

---

### 技术亮点总结（1 分钟）

"我觉得这个项目有几个值得关注的点：

**一、零配置启动**。`pnpm install` 装完所有依赖，填一个 API Key，`pnpm dev` 一条命令跑起来，不用 Docker，不用数据库。

**二、技能是可扩展的插件**。新增一种设备类型，只需要写一个 skill.yaml + knowledge.md + actions.py，不改核心代码。这是一个真正的插件架构。

**三、记忆是可读的文本**。所有偏好和历史都存成 Markdown 和 JSON，用户可以直接打开看、直接修改——这比黑盒数据库更透明，也更容易调试。

**四、支持本地 LLM**。配上 Ollama，整个系统可以完全离线运行，数据不出本地网络。"

## 六、常见问题 Q&A

**Q: 没有小米设备，能演示吗？**
A: 可以。后端正常启动，Dashboard 完整展示，AI 决策流在有设备触发事件时才会有数据。可以通过 `POST /api/devices/add` 手动添加一个演示设备。

**Q: LLM 贵吗，API 费用高吗？**
A: 决策只在传感器变化时触发，不是持续轮询。DeepSeek 按 token 计费，普通家庭日常使用每天费用约几分钱。

**Q: 和 Home Assistant 有什么区别？**
A: HA 是规则引擎 + 自动化脚本，需要用户手动配置规则。Anima 的目标是让 AI 自动决策，不需要用户写规则——更接近"硬件的 AI 助手"而非"智能家居控制中心"。

**Q: 为什么用 MQTT 而不是直接 HTTP？**
A: MQTT 是物联网领域的标准协议，天然支持发布/订阅模式，适合多设备同时上报传感器数据的场景，延迟低，后续接入更多设备类型也更方便。
